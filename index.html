BMP文件

图片是由一个个像素（pixel）排列构成的。对于黑白图像来说，每个像素只需要1bit，就可以创造出一副图像。

对于彩色的图片，每个像素则需要更多的数据来展现不同的颜色。例如，GIF格式支持8位色彩（8-bit color），即每个像素的大小为8bit。BMP,JPEG或者PNG格式则支持24位色彩。而事实上，BMP格式支持1，4，8，16，24，32位色。

本节课程序中首先讨论的是24位色的BMP文件。它的每个像素由R,G,B三种颜色组成，每个元素的取值范围都是[0x00, 0xff]，每种颜色都占8bit。在BMP文件中，除了包含每个像素的色彩信息，在文件的开头也有着一部分元数据，元数据中存储着图片的必要信息。下图所示的就是在BMP format 4.0，Windows 95 上使用的BMP文件的文件头，这也是这节课中我们要处理的信息。

关于文件头的具体含义可以参考BITMAPFILEHEADER和BITMAPINFOHEADER的官方文档。

对于这个定义下的BMP文件，每一行的文件所包含的信息大小，必须是4Byte的整数倍，如果不是4的整数倍，则需要用大小为1Byte的0x00补全，我们称这些1Byte的填充数据为padding。虽然padding不参与图片的构成，但是它会影响图片文件的大小，即biWidth的大小是不包含padding的，但是biSize的大小是要计算padding的。

Resize

在这个练习中，我们要讲图片放大整数倍。最直观的修改方式就是宽和高对应增加n倍，同时复制n次原像素，即可完成放大n倍的任务。但对于整个文件，放大之后会改变的变量有以下5个：

bf.bfSize：图片像素、padding加上文件头所占的大小
bi.biSizeImage：图片像素加padding所占的大小
bi.biWidth：图片宽度
bi.biHeight：图片高度
padding：padding的个数

在确定了放大后会改变的变量之后，我们要考虑的是如何实现复写像素的问题。在横向上复写很简单，使用简单的for循环将写入操作重复操作n便即可。但是对于纵向上的复写就要更复杂一些，这里需要理解指针和文件指针。

指针

指针是C语言中比较难以完全理解但同时功能非常强大的特性之一，因为它可以直接操作内存，完成一些花哨的技巧。

从定义上来说，指针是一个变量，它包含另一个变量的内存中的地址。 我们可以有一个指向任何变量类型的指针。在使用上一般有两个符号，&和*。&variable 是调用某个变量的地址，*variable 则是调用该变量所在地址的值。

假设我们想定义一个交换输入变量的值的函数swap()，定义如下。我们期待得到两个值交换的结果，然而程序实际运行后会发现，两个变量的值没有发生变化。原因是因为，当运行到主函数里面的子函数时，程序会单独开辟出一块新的内存来对这个程序做运算，此时ab的值都是存在新的地址中的。当我们在子函数中得到了结果，回到主函数时，子函数所用的内存块就被抛弃了，这也就导致了主函数的两个变量的值没有交换。

但如果我们使用指针,则可以实现两个变量交换的结果

第一章：从零基础到部署

Cloud9

cloud9是依托于AWS EC2实例的开发环境，致力于在服务器端运行和存储代码，免去开发者重复部署环境和更换变成地点的困扰。

Bundler

Bundler是Rails管理 gems的工具，它通过根目录下的Gemfile文件来生命和控制gems的版本，以保证在运行Rails工程的时候可以找到所有的依赖库，以及确保使用正确版本的gems。

在当与其他人共同开发同一个项目的时候，对方使用bundle install命令后，bundler 会找到Gemfile.lock文件并确保他们的安装环境中的gems和在你的环境中的gems的版本是一样的，这就避免了因版本升级导致的第三方库的依赖问题。

参考资料：https://bundler.io/rationale.html

在编译中学习使用HTTParty的时候，第一次运行Bundler的时候我没有在Gemfile文件中声明所用的gem的名字，而是通过命令行安装，这就导致了在编译Rails工程的时候一直无法找到这个依赖库，快把网上的文档翻烂了才在Bundler的官方文档里查出原因。整个过程大概浪费了1个小时。

Model-View-Controller (MVC)

Rails遵循MVC框架，它的设计理念是将数据、前端和后台控制分离，以保证整体应用的稳定运行。

  1. The browser issues a request for the /users URL.
  2. Rails routes /users to the index action in the Users controller.
  3. The index action asks the User model to retrieve all users (User.all).
  4. The User model pulls all the users from the database.
  5. The User model returns the list of users to the controller.
  6. The controller captures the users in the @users variable, which is passed to the index view.
  7. The view uses embedded Ruby to render the page as HTML.
  8. The controller passes the HTML back to the browser.

好的论证的五大原则

1、良好论证的结构性要求
2、论证之前提的相关性
3、论证之前提的可接受性
4、相关且可接受的前提对结论支持的充分性
5、论证对其最有力的批评的反驳，是否有效

为什么需要好的论证

1、好的论证帮助人们更好地做出决定
2、好的论证帮助人们遇到艰难的道德选择事，做出正确的决定，避开不良后果
3、好的论证使人加强现有的观念，或暴露现有的缺点
4、好的论证能提升生活中思考的谁准，更能使人接受你的观点
5、好的论证可以解决争执，使双方就道理而争论，而不是争论其他的

第一章：智力行为准则

本章内容小结

1、 遵守智力行为12条基本准则，来解决争议
2、 对要争议的问题持有可靠的观点，第一步必须承认自己可能会犯错，要随时准备接受新的观点
3、 表达观点时，小心避免语言及概念混乱，有的问题经常不恰当的和别的问题混在一起

十二条基本准则：

1.或缪原则 ：自己可能是错的
2.求真原则 ：认真对待对方的观点，查找对方观点中的亮点，允许对方就任何方面发表论证，以达到所能达到的最合理的论证（终极目标是发现真理）
3.清晰原则 ：仔细推敲用词，不能有语义混乱和观点混淆，并能够从不同的观点中清晰的区分出主次。
———————————以上三种是严肃讨论问题的逻辑基石———————————
4.举证原则 ：谁主张谁举证
5.善意原则 ：如果复述对方的论证，要认真表达，不得减弱其论证力，不得曲解对方原意
6.结构原则 ：结论至少有一个理由的支持，如果论证是规范性的，至少要拥有一个规范性前提。不使用相互冲突的推理，不使用与结论相矛盾的推理，不预先假设结论的真实性，不做不合逻辑的演绎推导。
7.相关原则 ：辩论中只能提出对结论有贡献的论点
8.接受原则 ：论点必须是一个有理智的成人可接受的
9.充分原则 ：论点必须是充分的
10.辩驳原则 ：驳斥对方时，应对其立场的可预见非难之处有所辩驳，不应回避其最有力之处
11.延迟判断原则 ：如果辩论不能支持任何一种立场，延迟判断
12.终结原则 ：如果不同立场的某一立场结构完整，使用着相关可接受推论，为结论所做了基础，并且包含关键性的反驳。除非有人可以继续提出这些条件没有满足，否则辩论结束。但如果随后还有异议，有权利重新开始辩论

第二章：什么是论证

本章内容小结

1、 了解论断和意见的区别，在辩论过程中谁主张谁举证
2、 公正的构建一个标准的论证，以做出合适的评判
3、 分清演绎论证和归纳论证的差异，以弄清不同结论的强弱效果
4、 知晓规范性论证的都有特质和规范性前提的重要性

论点即其他论断所支持的论断：

论证是一组陈述，其中的一个或几个陈述为结论提供支持，这些陈述成为前提。为前提提供论据支持的陈述，称为子前提，子前提使前提有了更强的说服力。
理性的反思证据后所获得的判断或观点，往往被成为结论。
通过好的论证得到的结论有前提支撑，没有证据支撑的结论只能算作“毫无根据的意见”。
论证和表达意见的区别在于，表达意见只提出了观点，但没有证据支撑。就比如说一个人只是简单的告诉我得相信什么，而不告诉我为什么要信。而论证是经得起论辩的。
人们常说：“每个人都有权利发表自己的观点。”这话没错，但问题不在于我们是否有发表的权力，而在于那些观点是否是值得接受的。
评判对方的看法，要集中在论证的质量上，看看对方是否提供了可靠的证据支撑。

谁主张谁举证：

让对方无凭无据的同意你的看法，或者推卸自己的举证责任，或称对方如果提不出相关证据就表示你的观点更可靠等，都违反了逻辑错误。

逃避举证的方法：

使用“没有理由相信xxx”，此时不承担举证责任。如果说“不相信xxx”，则需要对观点举证。

Something about Learning How to Learn

ThinkingMode

Focused Mode and Diffuse Mode

专注模式指当你全神贯注的学习或者试图理解一件事情的时候的大脑状态。而发散模式是一种放松的思考模式。

如图1中所示，把大脑比作一个弹珠机（pinball machine），专注模式中桔黄色的部分是已熟悉的思维模式（公式等），当你产生一个需要思考的问题时，思维会很快进入由几个神经元组成的回路，即会优先去在熟悉的领域找到类似的解决办法或概念。在找到这个解决办法后，就会按部就班的用熟悉的方法解决这个问题。

而在发散模式中思维会随性的在神经元之间跳跃，它不会着眼于某个特定的解决问题的办法或概念，但是会找到一个解决问题的出发点。

就目前神经学家所知，专注模式与发散模式只能单独存在，不能同时使用两种模式。这有点像一枚硬币，我们一次只能看到它的其中一面，但是却不能同时看到两面。

在学习一些困难的知识时，需要在这两种模式间转化，以消化理解新的内容。

Memory

在学习新的知识，尤其是抽象的概念时，需要不断地重复，使神经元的回路从微弱的联系逐渐变成坚实的回路。但在进行了一段时间的专注学习后，应当休息一下，使神经元回路更好的构建。

Working Memory and Long-term Memory

工作记忆，是大脑用来存放当前正在处理的内容的一部分记忆。它集中在大脑的前额叶皮质。
长期记忆，就像一个仓库，是存放已经记住的的内容的一部分记忆。

如果希望将工作记忆转化成长期记忆，则需要时间和重复的练习。有一种叫做“间隔重复”的方法，即不断重复你尝试记忆的内容，但重复的过程必须间隔开来。像《17天搞定GRE单词》里那种把重点放在不断的复习旧list的方法，也是一样的道理。

在睡前重温一下想要巩固记忆的内容，可能会有明显的效果。

Chunk

如果把学习到的知识点比作拼图的碎片的，那chunk则代表有数块拼图碎片块组成的一个拼图整体的局部。在学习过程中，把分散的信息组成一个又相互关联的大的信息碎块，更便于记忆。
组块是更紧凑的压缩信息的一种方式。

要注意Chunk和Chunk之间的联系。如果一个Chunk对你造成了认知负载（Cognitive Load），则把它拆成一个个的小Chunk，逐步完成。

  • 构建chunk时要专注，不要让无关的事情占用working Memory。
  • 要理解构成组块的概念，当不理解时构成的是无用的Chunk。
  • 复习，并实践。初次形成的Chunk如果没有经过反复的复习，则神经元之间的联系会减弱，即便是在课堂上听懂了的内容也会在之后遗忘。“In math and science related subjects, closing the book and testing yourself on whether you, yourself, can solve the problem you think you understand, will speed up your learning at this stage.”
  • 扩大思考的范围。重复练习相关或者不相关的问题，可以学到使用Chunk解决问题的时机。

Chunk是用来解决某个特定问题的，Context是决定在某个问题上使用哪一个Chunk。在学习中，可以采取自下而上（从问题入手）的方法或者自上而下（从大局big picture入手）的方法这两者的重合点就是 Context。在学习中，先对整体(big picture)有一个了解，再处理局部（chunk）问题，这样即使在最后有一些细节产生了遗漏，也不会对理解整个问题产生很大的影响。

Recall

在学习中，使用概念图（Concept Map）或者反复阅读的方法，都不如回想（Recall）的效果好。在学习完一个新的概念后，离开材料，在心中检索关键的概念，这是构建Chunk以及在Chunk之间建立联系的最好办法。

在常规的学习场所之外的地方回想，会帮助对所学内容的理解。因为学习者通常会把最开始接触材料的地方，当作潜意识中的提示。通过在不同物理环境中的回想，可以摆脱对固定场所的依赖。

在上课或者看完答案后就觉得自己理解了，是一种常见的能力幻想（Illusion of Competence），同样在书本上用高亮方式做标注，也是一种能力幻想，即误以为自己理解或记住了要学的内容。但在空白处写笔记总结关键概念则是一种好的学习方法。

为了对抗Illusion of Competence，需要在学习过程中不断的测试自己，Recall就是一种很好的测试方法。

Transfer

一个人如果积累了很多的Chunk，那么就像是他脑中存放了很多的神经回路。在遇到一个新的问题时，Chunk和Chunk之间可以产生相互联系，用旧的Chunk来解决新的问题，这种现象称作迁移（transfer）。在这种情况中，发散模式起到了很大的作用。

在解决问题有两种模式，一种是顺序性的一步步推理，类似于专注模式，需要逐步的有目的的解决一个个小问题；另一种是通过直觉，类似于发散模式，联系在专注模式中不会联系起来的观点。要记住直觉不永远是正确的，需要用专注模式验证。

Overlearning

当完全掌握一个知识后，再继续的针对这个内容进行训练的过程，即称作过度学习（Overlearning）。

过度学习会使行为自动化，即非常熟练，这对于对抗紧张情绪是非常有帮助的。

但研究表明，过度学习更多的情况下是对时间的浪费。就像是只通过锤子敲击钉子来学习木工，你可以很熟练的用锤子固定住钉子，以至于会误以为可以用锤子解决所有的问题。这就造成了Illusion of Competence。因为过度学习后的内容对学习者来说是『简单的』，而重复做简单的工作会使你误以为你掌握了这个领域里的全部内容，而实际上只掌握了简单的部分。

因此应该将更多的精力集中在难点上，即刻意训练 (deliberate practice) 。

Einstellung

思维定势（Einstellung）指已经存在的思维模式会阻碍更新更好想法的形成，很多时候需要使用发散模式去克服思维定势。

Interleaving

在学习一门新学科时，不仅要积累Chunk，还要知道如何选择和应用不同的Chunk，而最好的做法就是在需要不同技术和策略的问题以及情形中来回转换，即交替学习（Interleaving）。

同时学习不同的东西，比同一时间段内只学习一个东西要有效。因为这样能促进不同领域之间的组块间创造联系，提高创造性。

Tips

  • 将要记住的东西联想一副难忘的画面，越生动越有助于记忆。
  • 使用联想，把数字与年份或者其他熟悉的东西联系在一起。
  • 运动，使身体和神经元充满活性。
  • 使用比喻和类比(Metaphor and Analogy)进行学习。

Procrastination

拖延（熟悉的词
通常情况下，人们会在让人不舒服的事情上拖延。拖延和成瘾有相似点，它可以让你短暂地兴奋并从无聊的现实中解脱出来。

关注过程而非结果

应对拖延，可以在学习过程中把注意力集中在学习的过程，而非学习的结果上。在面对一项难以推进的任务时，应当着眼于自己对这项任务上花了多长时间，而不是今天要完成到什么结果，这样就不会因为面对可能难以完成的结果而产生负面情绪，有利于工作的进行。

改变固有的习惯

  • Tips
  1. 信号The Cue 找到会使自己进入拖延状态的激励并尽可能避免它，例如学习前关掉手机。
  2. 惯势Routine 改变那些会使自己进入拖延的日常习惯，尽量待在一个没有拖延诱发激励的新环境中，并且对新环境产生信赖，形成新的惯势。
  3. 奖励Reward 在完成了不拖延的目标后给自己正面的奖励。用这种方法可以促进自己形成新的习惯。
  4. 信念 The Belief 类似于正念。当事情变得棘手时，人们就会渴望回到那个更舒适的旧习惯去，而相信新的习惯能够奏效这一信念可以让你坚持下去。
  • eat your frogs first in the morning* 在早晨先完成困难的任务。
  • 做周计划和每日计划，每日计划最好在前一天晚上制定，这样在睡觉时潜意识会设法安排和完成任务。同时，将计划从WorkingMemory转移到纸上也能更好的利用working Memory了。
